# Musicbot – TeamSpeak Client Deployment

`easywi-teamspeak-client` is the CGo subprocess that speaks the TeamSpeak 3/6
client SDK on behalf of the musicbot. It is spawned by `easywi-teamspeak-bridge`
and communicates over stdin/stdout using the NDJSON protocol described in
[`docs/architecture/musicbot-teamspeak-external-bridge-protocol.md`](../architecture/musicbot-teamspeak-external-bridge-protocol.md).

---

## 1. Build Modes

Two mutually exclusive build modes exist.

### 1.1 Stub (default – no CGo)

```bash
go build ./agent/cmd/easywi-teamspeak-client/
```

The resulting binary compiles without any external libraries. Every `connect`
request returns `client_backend_required`, instructing the operator to supply a
real SDK library. Use this build for distribution in environments where the
proprietary SDK has not been licensed yet.

### 1.2 Real SDK (`ts3clientlib` – CGo)

```bash
# Prerequisites
apt-get install libopus-dev          # provides libopus.so / libopus.so.0
# Place libts3client.so somewhere on the server (see §3)

CGO_ENABLED=1 go build -tags ts3clientlib ./agent/cmd/easywi-teamspeak-client/
```

**Build-time dependencies**

| Library | Purpose | How to obtain |
|---------|---------|---------------|
| `libopus-dev` | Opus → PCM decode | `apt-get install libopus-dev` |
| `libts3client.so` | TeamSpeak 3/6 SDK | Register at <https://teamspeak.com/en/features/teamspeak-sdk/> and download the Linux SDK |

The CGo linker flags are `-ldl -lpthread` only. Both `libopus.so` and
`libts3client.so` are loaded at **runtime** via `dlopen`; they are not linked
at build time.

---

## 2. Required Files

After building, collect the following files before deploying:

| File | Required | Notes |
|------|----------|-------|
| `easywi-teamspeak-client` | Yes | The compiled binary (stub or ts3clientlib) |
| `libts3client.so` | Yes (ts3clientlib mode) | Proprietary – must be provided by the admin. Never commit to the repository. |
| `libopus.so.0` / `libopus.so` | Yes (ts3clientlib mode) | Usually installed system-wide via `libopus-dev` |
| `<identity>.ini` | Optional | TeamSpeak identity file. Generated by the client on first connect if omitted. |
| `sdk-resources/` | Optional | SDK resource files that ship alongside `libts3client.so` (if any). Place them in the same directory as `libts3client.so`. |

**Do not distribute `libts3client.so` or any SDK files.** The TeamSpeak SDK
license prohibits redistribution. Admins must obtain and place the library
themselves.

## 3. Installation

### 3.1 Recommended directory layout

```
/opt/easywi/musicbot/teamspeak-client/
├── easywi-teamspeak-client      # the binary
├── libts3client.so              # SDK library (admin-provided)
├── libopus.so.0 -> /usr/lib/x86_64-linux-gnu/libopus.so.0  # symlink or copy
└── identity.ini                 # optional; generated on first connect
```

The directory containing `libts3client.so` is automatically used as the SDK
resources directory (`ts3bridge_init` receives `dirname(libts3client.so)`).
Place any SDK companion files (e.g. `soundbackends/`) in the same directory.

### 3.2 Ownership and permissions

```bash
# Binary
install -m 0755 easywi-teamspeak-client /opt/easywi/musicbot/teamspeak-client/
chown musicbot:musicbot /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client

# SDK library (admin places here)
chown musicbot:musicbot /opt/easywi/musicbot/teamspeak-client/libts3client.so
chmod 0644 /opt/easywi/musicbot/teamspeak-client/libts3client.so

# Identity file (if pre-generated)
chown musicbot:musicbot /opt/easywi/musicbot/teamspeak-client/identity.ini
chmod 0600 /opt/easywi/musicbot/teamspeak-client/identity.ini
```

The binary must be **executable** (`0755`). The `libts3client.so` does **not**
need to be executable.

### 3.3 Library search path

`libopus.so.0` must be findable by `dlopen`. Options (choose one):

```bash
# Option A – system-wide (recommended)
apt-get install libopus-dev

# Option B – LD_LIBRARY_PATH in the bridge service unit
Environment=LD_LIBRARY_PATH=/opt/easywi/musicbot/teamspeak-client

# Option C – ldconfig
echo /opt/easywi/musicbot/teamspeak-client > /etc/ld.so.conf.d/easywi-ts.conf
ldconfig
```

### 3.4 Checksum verification

After placing the binary, record its checksum for integrity monitoring:

```bash
sha256sum /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client \
  > /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client.sha256
```

## 4. Runtime Configuration

The musicbot runtime reads its TeamSpeak connector settings from the musicbot
JSON configuration (typically managed via the admin panel). The relevant keys
are listed below.

### 4.1 Minimal configuration

```json
{
  "teamspeak": {
    "enabled": true,
    "backend_type": "external_client_bridge",
    "backend_path": "/opt/easywi/musicbot/teamspeak-bridge/easywi-teamspeak-bridge",
    "config": {
      "client_backend_type": "client_library",
      "client_library_path": "/opt/easywi/musicbot/teamspeak-client/libts3client.so"
    },
    "host": "ts3.example.com",
    "port": 9987,
    "channel_id": "1",
    "nickname": "Musicbot"
  }
}
```

### 4.2 Full reference

| JSON key | Type | Required | Description |
|----------|------|----------|-------------|
| `teamspeak.enabled` | bool | Yes | Enable TeamSpeak connector |
| `teamspeak.backend_type` | string | Yes | Must be `"external_client_bridge"` for the real SDK path |
| `teamspeak.backend_path` | string | Yes | Absolute path to `easywi-teamspeak-bridge` binary |
| `teamspeak.host` | string | Yes | TeamSpeak server hostname or IP |
| `teamspeak.port` | int | Yes | UDP port (default: `9987`) |
| `teamspeak.channel_id` | string | Yes | Numeric channel ID to join |
| `teamspeak.nickname` | string | Yes | Bot nickname displayed in TeamSpeak |
| `teamspeak.server_password` | string | No | Server password (never logged) |
| `teamspeak.channel_password` | string | No | Channel password (never logged) |
| `teamspeak.identity_path` | string | No | Path to TeamSpeak identity file (`.ini`) |
| `teamspeak.config.client_backend_type` | string | Yes | `"client_library"` to use `easywi-teamspeak-client` with real SDK |
| `teamspeak.config.client_library_path` | string | Yes | Absolute path to `libts3client.so` |
| `teamspeak.config.bridge_backend_type` | string | No | Alias for `client_backend_type` (either key is accepted) |

**Additional accepted aliases for `client_library_path`** (in lookup order):
`native_sdk_path`, `sdk_path`, `library_path`.

### 4.3 Process chain

```
musicbot runtime
  └─ easywi-teamspeak-bridge   (EASYWI_TS_BRIDGE=1, NDJSON on stdin/stdout)
       └─ easywi-teamspeak-client   (EASYWI_TS_CLIENT_LIB=1, NDJSON on stdin/stdout)
            └─ dlopen(libts3client.so)
                 └─ dlopen(libopus.so.0)
```

The bridge binary path (`backend_path`) must be `easywi-teamspeak-bridge`, not
the client binary. The bridge spawns `easywi-teamspeak-client` automatically
using the path it received in the connect request (`client_library_path` →
`backend_path` field in the bridge's connect JSON).

## 5. Validation

### 5.1 Smoke-test the binary directly

```bash
# Confirm the binary starts and responds to a status request
echo '{"id":"1","action":"status"}' | \
  EASYWI_TS_CLIENT_LIB=1 /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client
# Expected: {"id":"1","ok":true,"ready":false,"state":"disconnected"}
```

### 5.2 Confirm SDK loads at connect time

```bash
echo '{"id":"2","action":"connect","host":"ts3.example.com","port":9987,
       "nickname":"test","backend_path":"/opt/easywi/musicbot/teamspeak-client/libts3client.so"}' | \
  EASYWI_TS_CLIENT_LIB=1 /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client
# Expected on success: {"id":"2","ok":true,"ready":true,"state":"connected","client_id":"..."}
# Expected without SDK: {"id":"2","ok":false,"error":"connect: failed to load TeamSpeak SDK ..."}
```

### 5.3 Agent-level validation via Symfony console

```bash
php bin/console musicbot:teamspeak_backend install --musicbot-id=<id>
php bin/console musicbot:teamspeak_backend status  --musicbot-id=<id>
php bin/console musicbot:teamspeak_backend repair  --musicbot-id=<id>
```

Expected `capability_status` after a successful connect with the real SDK:
`ready`.

### 5.4 Full E2E test

```bash
# With the easywi-ts-e2e-helper (no real SDK required — CI mode)
scripts/musicbot-live-e2e.sh

# With the real SDK (requires libts3client.so and a TS3 server)
MUSICBOT_E2E_RUN_TEAMSPEAK=1 \
MUSICBOT_E2E_TS_HOST=ts3.example.com \
MUSICBOT_E2E_TS_CHANNEL_ID=1 \
MUSICBOT_E2E_TS_CLIENT_BACKEND_PATH=/opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client \
  scripts/musicbot-live-e2e.sh
```

All checks must pass. `frames_sent > 0` requires `ffmpeg` in `PATH` for WAV
decoding; on GitHub Actions `ubuntu-latest` it is pre-installed.

## 6. Security Constraints

- **No secrets in logs.** `server_password` and `channel_password` are masked
  in every log line by `mask()` before any output. They are passed to the SDK
  as raw C pointers, never formatted into strings. Verify with:
  `grep -i password /path/to/bridge.log` — must return nothing.

- **No automatic download.** Neither the runtime nor the bridge fetches
  `libts3client.so` automatically. The admin must place the file manually.
  Any automation that downloads SDK binaries from unverified sources violates
  the TeamSpeak SDK license and introduces supply-chain risk.

- **No SinusBot or TS3AudioBot binaries.** The bridge validator rejects any
  `backend_path` whose filename contains `sinusbot` or `ts3audiobot`
  (case-insensitive). Attempting to use those binaries returns an error and
  the agent will not start.

- **No placeholder as ready.** A stub binary (built without `-tags ts3clientlib`)
  always returns `capability_status: client_backend_required`. It never claims
  `ready`. Do not override this check; the placeholder exists to guide
  operators, not as a fallback for production use.

- **No production servers for testing.** Use an isolated, dedicated TeamSpeak
  server instance for E2E tests. Never connect a test bot to a production guild
  or server.

- **Binary exec bit required.** The bridge validates that `easywi-teamspeak-client`
  has the executable bit set before spawning it. Deploying without `chmod +x`
  will cause an explicit error rather than a silent failure.

## 7. Troubleshooting

### `libts3client.so` not found / failed to load

```
connect: failed to load TeamSpeak SDK from /opt/... — ensure libts3client.so and libopus.so are installed
```

**Check:**
1. Does the file exist at the configured `client_library_path`?
2. Is it readable by the musicbot process user?
3. Was the binary built with `-tags ts3clientlib`? (The stub build ignores the path entirely.)

```bash
ls -la /opt/easywi/musicbot/teamspeak-client/libts3client.so
file /opt/easywi/musicbot/teamspeak-client/libts3client.so
# Expected: ELF 64-bit LSB shared object
```

---

### `libopus.so` not found

```
connect: failed to load TeamSpeak SDK ... (dlopen: libopus.so.0: cannot open shared object file)
```

**Fix:**

```bash
apt-get install libopus-dev
# or add the library directory to LD_LIBRARY_PATH before starting the bridge
```

---

### Binary not executable

```
validate_teamspeak_bridge_path: file is not executable
```

**Fix:**

```bash
chmod +x /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client
```

---

### Wrong `backend_path` (bridge vs client confusion)

The `teamspeak.backend_path` in the musicbot config must point to
`easywi-teamspeak-bridge`, **not** `easywi-teamspeak-client`. The bridge then
spawns the client. If the client binary is set as `backend_path`, the connect
request will fail because the client does not speak the bridge protocol.

**Verify:**

```bash
file /path/to/configured/backend_path
strings /path/to/configured/backend_path | grep EASYWI_TS_BRIDGE
# Should print: EASYWI_TS_BRIDGE
```

---

### Wrong channel ID

```
join_channel: channel_id "lobby" is not a valid numeric channel ID
```

TeamSpeak channel IDs are integers, not names. Find the numeric ID in the
TeamSpeak client: right-click the channel → *Channel Info* → note the ID.

---

### Server password rejected

The connect call returns `ok: false` with a generic connection-refused error.
Because passwords are masked, the exact value is not echoed. Double-check the
`server_password` value in the musicbot config. Temporarily connect with an
unpassworded test server to confirm the binary and SDK work correctly before
adding credential complexity.

---

### `capability_status` stays `client_backend_required`

The bridge started but the client binary did not load the SDK. Possible causes:

1. Binary built without `-tags ts3clientlib` (stub build) — rebuild with the tag.
2. `client_library_path` is wrong or the file is missing — see §7.1 above.
3. `client_backend_type` is not set to `"client_library"` in the musicbot config.

```bash
# Quick check: run the client binary standalone
echo '{"id":"1","action":"status"}' | \
  EASYWI_TS_CLIENT_LIB=1 /opt/easywi/musicbot/teamspeak-client/easywi-teamspeak-client
# Stub:    {"id":"1","ok":true,"ready":false,"state":"disconnected"}  ← same either way
# Connect and check for "client_backend_required" in the connect response
```

---

### `frames_sent` stays `0`

Audio frames reach the bridge but the SDK is not transmitting them to the server.

**Checklist:**
- Is `ffmpeg` in `PATH`? The AudioPipeline requires it to decode WAV/MP3 to raw PCM.
- Is the bot actually joined to a channel? Check `current_channel_id` in the status response.
- Is the TeamSpeak server accepting audio? Check TS3 server logs for codec errors.
- Is the bot muted server-side? (TeamSpeak server-side mute silently drops audio.)

```bash
which ffmpeg || echo "ffmpeg missing — install ffmpeg"
```
